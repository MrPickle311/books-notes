Java Concurrency in Practice

Dogłębna analiza: Rozdział 3 - Współdzielenie Obiektów

3.1. Widoczność (Visibility)

W jednowątkowym środowisku zapis i odczyt zmiennej jest przewidywalny. W środowisku wielowątkowym brak synchronizacji sprawia, że wątek czytający może nie zobaczyć zmian wprowadzonych przez wątek piszący. Dowodzi tego klasyczny przykład NoVisibility.

// Listing 3.1. Sharing Variables without Synchronization. Don't Do this.
public class NoVisibility {
    private static boolean ready;
    private static int number;

    private static class ReaderThread extends Thread {
        public void run() {
            while (!ready)
                Thread.yield();
            System.out.println(number);
        }
    }

    public static void main(String[] args) {
        new ReaderThread().start();
        number = 42;
        ready = true;
    }
}

Program z Listingu 3.1 może wydrukować 42, może działać w nieskończoność (nie widząc zmiany ready), a z powodu zmiany kolejności (reordering) instrukcji przez kompilator/procesor, może wydrukować 0.

3.1.1. Nieświeże dane (Stale Data)

Kiedy wątek czyta zmienną bez synchronizacji, może otrzymać przestarzałą wartość. Listing 3.2 pokazuje klasę nienależącą do bezpiecznych, a Listing 3.3 to jej naprawiona wersja.

// Listing 3.2. Non-thread-safe
// Mutable Integer Holder.
@NotThreadSafe
public class MutableInteger {
    private int value;

    public int get() { 
        return value; 
    }
    public void set(int value) { 
        this.value = value; 
    }
}
// Listing 3.3. Thread-safe 
// Mutable Integer Holder.
@ThreadSafe
public class SynchronizedInteger {
    @GuardedBy("this") 
    private int value;

    public synchronized int get() { 
        return value; 
    }
    public synchronized void set(int v) { 
        this.value = v; 
    }
}

Uwaga: Samo dodanie synchronized do metody set w Listingu 3.2 to za mało! Metoda get również musi być zsynchronizowana, aby zablokować odczyt "nieświeżych" danych z cache.

3.1.2. Nieatomowe operacje 64-bitowe

Dla zmiennych 32-bitowych Java gwarantuje, że odczytamy chociaż jakąś spójną (nawet jeśli nieświeżą) wartość. Zmienne 64-bitowe (long i double) nie mają tej gwarancji, jeśli nie są volatile. Maszyna wirtualna może zapisać je w dwóch 32-bitowych operacjach, prowadząc do "rozerwania" (tearing) - wątek czytający otrzyma połowę nowej i połowę starej wartości.

3.1.3. Blokowanie i widoczność (Locking and Visibility)

Blokady nie służą tylko do wzajemnego wykluczania (mutual exclusion). Zapewniają one także widoczność pamięci. Zobrazowano to na słynnym Rysunku 3.1.

Schemat pokazujący, że wątek B, wchodząc do tego samego bloku synchronized, gwarantuje sobie widoczność nie tylko zmiennej x, ale też y, które zostało przypisane jeszcze przed blokadą przez wątek A.

3.1.4. Zmienne Volatile

Słowo volatile jest słabszą formą synchronizacji. Wymusza pobieranie i zapisywanie zmiennej bezpośrednio w pamięci głównej.

// Listing 3.4. Counting Sheep.
volatile boolean asleep;

while (!asleep)
    countSomeSheep();

volatile nadaje się świetnie do flag (jak wyżej), ale nie gwarantuje atomowości (np. inkrementacji). Należy go używać, gdy stan zmiennej nie zależy od jej poprzedniej wartości.

3.2. Publikacja i Ucieczka (Publication and Escape)

Publikacja to udostępnienie obiektu na zewnątrz. Ucieczka (escape) ma miejsce wtedy, gdy obiekt staje się publiczny przedwcześnie.

// Listing 3.5. Publishing an Object.
public static Set<Secret> knownSecrets;

public void initialize() {
    knownSecrets = new HashSet<Secret>();
}
// Listing 3.6. Allowing Internal Mutable State to Escape. Don't Do this.
class UnsafeStates {
    private String[] states = new String[] { "AK", "AL" };
    public String[] getStates() { return states; } // Zła praktyka: tablice są modyfikowalne!
}
// Listing 3.7. Implicitly Allowing the this Reference to Escape. Don't Do this.
public class ThisEscape {
    public ThisEscape(EventSource source) {
        source.registerListener(
            new EventListener() {
                public void onEvent(Event e) {
                    doSomething(e); // Wątek zewnętrzny może wywołać to przed końcem konstruktora!
                }
            }
        ); 
    }
}

3.2.1. Praktyki bezpiecznej konstrukcji

Obejściem problemu z Listingu 3.7 jest ukrycie wywołania startującego w statycznej metodzie fabrykującej, by nie pozwolić na ucieczkę this przed pełną inicjalizacją.

// Listing 3.8. Using a Factory Method to Prevent the this Reference from Escaping.
public class SafeListener {
    private final EventListener listener;

    private SafeListener() {
        listener = new EventListener() {
            public void onEvent(Event e) { doSomething(e); }
        };
    }

    public static SafeListener newInstance(EventSource source) {
        SafeListener safe = new SafeListener();
        source.registerListener(safe.listener); // Pełne bezpieczeństwo
        return safe;
    }
}

3.3. Ograniczenie do wątku (Thread Confinement)

Dostęp do modyfikowalnych danych z jednego wątku to prosty sposób na zapewnienie bezpieczeństwa (tzw. zamykanie w wątku).

3.3.1. Ad-hoc Thread Confinement

Podejście nieformalne, opierające się tylko na założeniach programisty (kruche, niepolecane).

3.3.2. Ograniczenie do stosu (Stack Confinement)

Korzystanie z lokalnych zmiennych metody, z których obiekty referencyjne nigdy nie "uciekają".

// Listing 3.9. Thread Confinement of Local Primitive and Reference Variables.
public int loadTheArk(Collection<Animal> candidates) {
    SortedSet<Animal> animals;
    int numPairs = 0;
    Animal candidate = null;

    // Zmienna animals zablokowana do lokalnego stosu wywołań.
    animals = new TreeSet<Animal>(new SpeciesGenderComparator());
    animals.addAll(candidates);
    // ... pętla for, która nie upublicznia 'animals'
    return numPairs;
}

3.3.3. ThreadLocal

Mechanizm dający wątkowi odrębną, własną kopię danej zmiennej (często po to, aby zapobiec konieczności stosowania blokad).

// Listing 3.10. Using ThreadLocal to Ensure thread Confinement.
private static ThreadLocal<Connection> connectionHolder
    = new ThreadLocal<Connection>() {
        public Connection initialValue() {
            return DriverManager.getConnection(DB_URL);
        }
};

public static Connection getConnection() {
    return connectionHolder.get();
}

3.4. Niezmienność (Immutability)

Obiekty całkowicie niezmienne są bezpieczne i niewrażliwe na błędy współbieżności.

// Listing 3.11. Immutable Class Built Out of Mutable Underlying Objects.
@Immutable
public final class ThreeStooges {
    private final Set<String> stooges = new HashSet<String>();

    public ThreeStooges() {
        stooges.add("Moe");
        stooges.add("Larry");
        stooges.add("Curly");
    }
    public boolean isStooge(String name) {
        return stooges.contains(name);
    }
}

3.4.1. Pola Finalne (Final Fields)

Użycie słowa kluczowego final jest fundamentalne, ponieważ nadaje tzw. bezpieczeństwo inicjalizacji (Initialization Safety). Programy powinny maksymalizować wykorzystanie final tam, gdzie zmienność nie jest wymagana.

3.4.2. Użycie Volatile do publikacji niezmiennych obiektów

Często chcemy zmienić zbiór powiązanych wartości atomowo bez użycia rygorystycznych blokad. Możemy wtedy zbudować niezmienny kontener i udostępnić go używając zmiennej volatile.

// Listing 3.12. Immutable Holder for Caching a Number and its Factors.
@Immutable
class OneValueCache {
    private final BigInteger lastNumber;
    private final BigInteger[] lastFactors;

    public OneValueCache(BigInteger i, BigInteger[] factors) {
        lastNumber = i;
        lastFactors = Arrays.copyOf(factors, factors.length);
    }

    public BigInteger[] getFactors(BigInteger i) {
        if (lastNumber == null || !lastNumber.equals(i))
            return null;
        else
            return Arrays.copyOf(lastFactors, lastFactors.length);
    }
}
// Listing 3.13. Caching the Last Result Using a Volatile Reference to an Immutable Holder Object.
@ThreadSafe
public class VolatileCachedFactorizer implements Servlet {
    private volatile OneValueCache cache = new OneValueCache(null, null);

    public void service(ServletRequest req, ServletResponse resp) {
        BigInteger i = extractFromRequest(req);
        BigInteger[] factors = cache.getFactors(i);
        if (factors == null) {
            factors = factor(i);
            cache = new OneValueCache(i, factors); // Publikacja nowego obiektu
            encodeIntoResponse(resp, factors);
        }
    }
}

3.5. Bezpieczna Publikacja (Safe Publication)

Kiedy zechcesz opublikować stan modyfikowalny, standardowa publikacja (jak poniżej) całkowicie narusza bezpieczeństwo.

// Listing 3.14. Publishing an Object without Adequate Synchronization. Don't Do this.
// Niewłaściwa (niebezpieczna) publikacja
public Holder holder;

public void initialize() {
    holder = new Holder(42);
}

3.5.1. Niewłaściwa publikacja (Improper Publication: When Good Objects Go Bad)

W wyniku optymalizacji pamięci iny wątek może zobaczyć, że obiekt holder ma przypisaną referencję, ale jego wewnętrzne pola mogą być niewidoczne w pełni zainicjowane! Ten błąd z Listingu 3.14 spowodowałby katastrofę w poniższej klasie:

// Listing 3.15. Class at Risk of Failure if Not Properly Published.
public class Holder {
    private int n;

    public Holder(int n) { this.n = n; }

    public void assertSanity() {
        if (n != n)
            throw new AssertionError("This statement is false.");
    }
}

3.5.2. Niezmienne obiekty i bezpieczeństwo inicjalizacji

Gdyby pola w obiekcie Holder były oznaczone jako final (obiekt byłby immutable), mechanizm modelu pamięci Javy (Initialization Safety) zagwarantowałby prawidłowe odczytanie stanu po konstrukcji, nawet gdy sam proces publikacji w 3.14 odbyłby się w wyniku wyścigu danych (data race).

3.5.3. Idiomy bezpiecznej publikacji (Safe Publication Idioms)

Dla obiektów modyfikowalnych, aby zapewnić współbieżne bezpieczeństwo podczas ich udostępniania innym wątkom, publikacja i stan obiektu muszą być widoczne dla obserwatora równocześnie. Należy to zrealizować poprzez:

  • Inicjalizację referencji wewnątrz bloków statycznych.
  • Przechowanie referencji w polu klasy zdefiniowanym jako volatile lub AtomicReference.
  • Przechowanie referencji jako pole final.
  • Użycie wewnętrznego zabezpieczenia przy zapisie (np. umieszczenie obiektu w zsynchronizowanej kolekcji czy `ConcurrentHashMap`).

3.5.4. Obiekty efektywnie niezmienne (Effectively Immutable Objects)

Jeśli opublikujesz obiekt bezpiecznie za pomocą powyższych idiomów, ale nie planujesz go później modyfikować w kodzie (np. wrzucasz obiekt Date do kolekcji bez zamiaru jego modyfikacji po publikacji), obiekt taki nazywamy efektywnie niezmiennym. Traktujemy go jak immutable i nie wymaga on dalszej synchronizacji po fazie bezpiecznej publikacji.

3.5.5. Obiekty modyfikowalne (Mutable Objects)

Jeśli opublikowany obiekt na pewno będzie zmieniany w przyszłości, to bezpieczna publikacja 3.5.3 chroni nas tylko i wyłącznie na moment przekazania tego obiektu (aby nikt nie odczytał w połowie zainicjalizowanego bytu). Każdą następną operację modyfikowania czy odczytu należy dodatkowo osłonić poprzez synchronizację np. przy pomocy słowa kluczowego synchronized lub własnej blokady Lock.

3.5.6. Bezpieczne współdzielenie obiektów (Sharing Objects Safely) - Reguły

Autorzy zamykają rozdział, porządkując wszystkie te metody w 4 fundamentalnych wytycznych projektowych (złotych zasadach JCiP):

1. Ograniczone do wątku (Thread-confined) Obiekt w całości znajduje się we władaniu wyłącznego dla siebie wątku.
2. Współdzielone, tylko do odczytu (Shared read-only) Obejmuje obiekty w pełni niezmienne oraz efektywnie niezmienne. Wolne od blokad.
3. Współdzielone, bezpieczne (Shared thread-safe) Obiekty realizujące w sobie odpowiednie założenia synchronizacyjne (np. java.util.concurrent).
4. Chronione (Guarded) Dostęp osłonięty jest poprzez konkretną blokadę narzuconą na kod kliencki lub klasę bazową.
