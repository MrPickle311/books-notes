Sieci w Świecie Kontenerów

Narzędzia takie jak Docker i Kubernetes nie tworzą żadnej własnej, czarnej magii sieciowej. Są one po prostu potężnymi orkiestratorami, które automatycznie używają linuksowych klocków (namespaces, veth, iptables, eBPF) tysiące razy na sekundę, aby połączyć ze sobą dynamicznie zmieniające się aplikacje.

13.1. Modele sieciowe Dockera

Kiedy uruchamiasz kontener na pojedynczej maszynie, Docker pozwala Ci wybrać poziom "izolacji" sieciowej. Istnieją 3 główne sterowniki (drivers).

Bridge (Domyślny)

--network bridge

Kontener 1
(netns)
Kontener 2
(netns)
docker0 (Linux Bridge)

Pełna izolacja. Docker używa własnego Namespace, tworzy kable Veth i wpina je do wirtualnego switcha docker0. By wyjść na świat, używa NAT (Masquerading).

Host

--network host

Kontener dzieli netns z Hostem!
eth0 (Fizyczna Karta)

Brak izolacji! Kontener nie dostaje swojego IP. Używa IP i portów gospodarza. Jeśli aplikacja nasłuchuje na porcie 80, zajmie port 80 całego serwera. Ekstremalnie szybkie (brak narzutu na NAT).

None

--network none

Kontener z lo (localhost)

Kontener dostaje swój Namespace sieciowy, ale Docker nie tworzy dla niego żadnych kabli veth. Całkowicie odcięty od świata. Używany do super-bezpiecznych zadań offline.

13.2. CNI: Jak Kubernetes łączy węzły (Flannel vs Calico)

Docker załatwia sprawę na jednym serwerze. Kubernetes to klaster dziesiątek serwerów (Node'ów). Kubernetes sam w sobie nie wie, jak połączyć Poda z Node'a 1 z Podem z Node'a 2. Zrzuca to zadanie na wtyczki CNI (Container Network Interface).

Zasada K8s: "Każdy Pod musi móc połączyć się z każdym Podem bez użycia NAT-u."

1. Sieć Nakładkowa (Overlay / VXLAN)

Wtyczki takie jak Flannel budują wirtualną sieć używając VXLAN (patrz Rozdział 4 i 6). Pakiety z Podów są w locie pakowane w koperty UDP i przesyłane do innego serwera.

Pod A (10.244.1.2)
Node 1 (192.168.0.10)
Pod B (10.244.2.5)
Node 2 (192.168.0.11)
FIZYCZNA SIEĆ (L3)
IP: .10 → .11 UDP/VXLAN Pod A → Pod B
Uwaga na MTU! Enkapsulacja zjada 50 bajtów pasma. Interfejsy w Podach muszą mieć mniejsze MTU (np. 1450).

2. Sieć Natywna + Routing BGP (Calico)

Wtyczki takie jak Calico odrzucają VXLAN (brak enkapsulacji = czysta prędkość). Jak to działa? Każdy Linux na Node zamienia się w mikro-router z odpalonym protokołem BGP!

BGP: "Mam sieć 10.244.1.0/24!"
Pod A (10.244.1.2)
Node 1 (Mikro-Router)
Pod B (10.244.2.5)
Node 2 (Mikro-Router)
CZYSTE IP
Zaleta: Najwyższa możliwa wydajność. Pakiet nie jest modyfikowany. System chmurowy dowiaduje się od BGP, by dopisać nowe Pody do fizycznych tabel routingu.

13.3. Kube-proxy (Wielka Iluzja ClusterIP)

Jeśli jest jedna rzecz, którą musisz zrozumieć w Kubernetesie z perspektywy sieci, to jest to zasada działania Service (typu ClusterIP).

Pody ciągle umierają, a ich adresy IP się zmieniają. Dlatego wprowadzono Serwisy (ClusterIP) – stabilne, wirtualne adresy IP przypisane "na zawsze". Sęk w tym, że ClusterIP to IP-widmo. Nie jest przypisane do ŻADNEJ karty sieciowej na żadnym serwerze w klastrze! Pingowanie ClusterIP zazwyczaj nie działa.

13.4. DNS w Kubernetes i pułapka ndots:5

Rozwiązywanie nazw wewnątrz klastra (DNS) jest realizowane przez wbudowany serwer (np. CoreDNS). Z racji tego, że pody muszą łatwo komunikować się z innymi serwisami w tej samej lub innych przestrzeniach nazw (np. pisząc po prostu baza-danych zamiast baza-danych.namespace.svc.cluster.local), Kubernetes stosuje bardzo agresywny mechanizm wyszukiwania domen.

Czym jest ndots:5 i jak degraduje wydajność?

Domyślnie w pliku /etc/resolv.conf każdego Poda zdefiniowana jest opcja options ndots:5. Informuje ona bibliotekę resolv systemu operacyjnego, że jeśli odpytywana nazwa domeny zawiera mniej niż 5 kropek, system ma najpierw próbować dokleić do niej wszystkie zdefiniowane lokalne sufiksy wyszukiwania.

Jeśli Twoja aplikacja próbuje odpytać zewnętrzną domenę, np. api.stripe.com (która ma 2 kropki < 5), wyszukiwanie przebiega następująco:

1. api.stripe.com.default.svc.cluster.local. → NXDOMAIN (Brak)
2. api.stripe.com.svc.cluster.local. → NXDOMAIN (Brak)
3. api.stripe.com.cluster.local. → NXDOMAIN (Brak)
4. api.stripe.com. → SUCCESS (Dopiero teraz!)

Każde jedno odpytanie o domenę zewnętrzną generuje 4 zapytania do CoreDNS. Może to drastiocznie przeciążyć serwer DNS w klastrze, zwiększyć opóźnienia i powodować losowe timeouty rozwiązywania nazw.

Jak to naprawić?

  • Kropka na końcu: Wpisuj w kodzie domeny bezwzględne, dodając kropkę na końcu, np. api.stripe.com.. System od razu zapyta o domenę główną, omijając lokalne wyszukiwanie.
  • Custom spec.dnsConfig: Możesz nadpisać wartość ndots dla konkretnego wdrożenia (Deployment) w Kubernetes do ndots:1, o ile aplikacja nie korzysta ze skróconych nazw lokalnych:
spec:
  containers:
  - name: my-app
    image: my-app:latest
  dnsConfig:
    options:
    - name: ndots
      value: "1"

Stworzono dla Inżynierów Systemów Rozproszonych.
