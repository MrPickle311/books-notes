Rozdział 11

Linuksowy Przybornik DSE (CLI)

Kod aplikacji często mówi "Connection Timeout" albo "Unknown Host", ale nie mówi *dlaczego*. Być może zablokowany jest port? Być może padł DNS? A może pakiety gubią się gdzieś po drodze w chmurze AWS? Poznaj narzędzia, które otworzą Ci oczy na to, co dzieje się w systemie operacyjnym.

11.1. Weryfikacja DNS i routingu (dig, mtr)

Jeśli Twoja usługa rzuca błędem połączenia, zawsze najpierw sprawdź rozwiązanie nazwy (DNS), a potem trasę pakietów. Przestań używać `ping` do sprawdzania trasy – jest przestarzały i często blokowany przez firewalle.

dig (Domain Information Groper)

Zastępuje przestarzałe `nslookup`. Służy do precyzyjnego sprawdzania rekordów DNS i czasu życia (TTL).

bash - dig
user@server:~$ dig google.com +noall +answer ; <<>> DiG 9.16.1-Ubuntu <<>> google.com +noall +answer ;; global options: +cmd google.com. 112 IN A 142.250.185.206 user@server:~$ _

Tip DSE: Zwracaj uwagę na TTL (tu: 112 sekund). Jeśli zmieniasz IP swojej bazy danych, przez 112 sekund klienci mogą nadal próbować łączyć się na stary adres, mimo Twojej aktualizacji rekordu!

mtr (My Traceroute)

Łączy funkcjonalność `ping` i `traceroute`. Ciągle odpytuje wszystkie routery po drodze do celu. Pozwala natychmiast wychwycić, gdzie w sieci "gubią się" pakiety.

bash - mtr
user@server:~$ mtr -n -c 10 8.8.8.8 HOST: server-prod1 Loss% Snt Last Avg Best Wrst StDev 1.|-- 10.0.0.1 (VPC Gateway) 0.0% 10 0.2 0.2 0.1 0.3 0.1 2.|-- 192.168.10.5 0.0% 10 1.5 1.4 1.2 1.6 0.2 3.|-- isp-router.zly.com (45.x) 80.0% 10 150.2 145.0 45.0 200.0 30.5 4.|-- 8.8.8.8 0.0% 10 15.3 15.5 14.2 16.8 0.8 user@server:~$ _

Tip DSE: Widać wyraźnie, że u operatora na 3-cim "skoku" (hop) mamy 80% Packet Loss (gubienia pakietów). To znaczy, że Twój serwer działa poprawnie, ale infrastruktura pomiędzy Tobą a docelowym API niedomaga (np. przez Bufferbloat). Omiń ten link lub zgłoś to adminom.

11.2. Badanie portów i połączeń (ss, lsof)

"Connection refused" albo "Address already in use". Klasyka gatunku. Zanim zrestartujesz serwer, musisz sprawdzić, kto "trzyma" dany port lub z kim serwer prowadzi najwięcej sesji. Zapomnij o starym `netstat`, używaj `ss`.

ss -tulpn

Szybki zrzut wszystkich nasłuchujących portów (tcp, udp, listening, process, numeric).

Netid State Recv-Q Send-Q Local Address:Port Peer tcp LISTEN 0 4096 127.0.0.1:5432 0.0.0.0:* users:(("postgres",pid=891)) tcp LISTEN 0 128 0.0.0.0:8080 0.0.0.0:* users:(("java",pid=1024))
Zwróć uwagę na 127.0.0.1 (Localhost - bezpieczne) vs 0.0.0.0 (Otwarte na wszystkie interfejsy sieciowe, czyli na świat!).

lsof -i

List Open Files. W Linuksie gniazdo sieciowe to po prostu plik. LSOF pozwala szybko znaleźć "kto zajmuje mój port".

user@server:~$ sudo lsof -i :8080 COMMAND PID USER FD TYPE DEVICE SIZE/OFF NODE NAME node 5432 root 18u IPv4 42111 0t0 TCP *:http-alt (LISTEN)
Chcesz odpalić apkę w Javie na porcie 8080, ale wali błędem? Uruchom lsof -i :8080 i od razu zobaczysz, że jakiś zabłąkany proces Node.js (PID 5432) go okupuje. Wpisujesz kill 5432 i po problemie.

11.3. Narzędzia warstwy L7 (curl, nc)

Ping dochodzi, port jest otwarty, a serwis nadal nie działa? Musimy wejść w warstwę siódmą, przyjrzeć się certyfikatom TLS i nagłówkom HTTP.

curl -v (Szwajcarski Scyzoryk Web Deva)

Flaga -v (verbose) otwiera czarną skrzynkę. Zamiast samego payloadu, widzisz cały proces L4 i L7.

user@server:~$ curl -v https://api.mojsystem.pl/health * Trying 192.168.1.5:443... * Connected to api.mojsystem.pl (192.168.1.5) port 443 (#0) * ALPN, offering h2 * ALPN, offering http/1.1 * SSL connection using TLSv1.3 / TLS_AES_256_GCM_SHA384 * Server certificate: * subject: CN=api.mojsystem.pl * start date: Jan 1 00:00:00 2024 GMT * expire date: Jan 1 00:00:00 2024 GMT (EXPIRED!) curl: (60) SSL certificate problem: certificate has expired user@server:~$ _
Co tu widać? Połączenie TCP (L4) nawiązało się poprawnie (Port 443 jest otwarty). Jednak aplikacja by nie zadziałała. Dlaczego? Bo curl -v wypluł, że certyfikat TLS wygasł. Przeglądarka wyświetliłaby błąd "Your connection is not private", a serwer w Javie rzuciłby SSLHandshakeException.

nc -zv (Netcat)

Ping blokowany przez firewall? Potrzebujesz sprawdzić, czy przepuści Cię do bazy PostgreSQL? Użyj Netcata. Flaga -z (Zero-I/O) tylko zestawia TCP handshake i natychmiast rozłącza, nie wysyłając payloadu. -v (Verbose) pisze, czy się udało.

user@server:~$ nc -zv 10.0.0.5 5432 Connection to 10.0.0.5 5432 port [tcp/postgresql] succeeded!

11.4. Packet Sniffing (tcpdump, Wireshark)

Gdy serwer i aplikacja milczą, a Ty chcesz "podsłuchać" dokładnie, jakie bajty wchodzą i wychodzą z Twojej maszyny. tcpdump to ostateczna broń na dziwne bugi (np. niewidzialne zrywanie połączeń przez sprzętowy NAT, pakiety ICMP czy sprawdzanie ramek VLAN).

Jak to działa? (Promiscuous Mode)

Zwykle karta sieciowa odrzuca pakiety, które nie są zaadresowane do jej adresu MAC. `tcpdump` uruchamia interfejs w trybie "promiscuous", przechwytując kopię każdego pakietu zanim ten trafi do zapory systemowej (iptables) czy aplikacji (L7).

# Przechwyć wszystko z portu 80 i zapisz do pliku
sudo tcpdump -i eth0 port 80 -w zrzut.pcap

Karta Sieciowa

tcpdump
Wireshark (Analiza pliku .pcap)

Czytanie "krzaczków" wyrzucanych przez `tcpdump` w konsoli jest trudne. Standardową praktyką inżyniera jest zrzucenie ruchu z serwera do pliku (flaga -w plik.pcap), ściągnięcie go na swój komputer (np. przez `scp`) i otwarcie w graficznym programie Wireshark. Wireshark pięknie rozkoduje każdy bit: pokaże numery sekwencyjne TCP, ramki HTTP/2, a nawet retransmisje zaznaczy na czerwono!

11.5. Zarządzanie siecią hosta (Pakiet iproute2)

Jeśli wciąż wpisujesz ifconfig żeby sprawdzić IP, albo route -n by zobaczyć trasy... przestań. Te narzędzia są zdeprecjonowane (deprecated) od ponad dekady w systemach Linux. Nowym standardem jest potężne narzędzie `ip`.

Sprawdzenie Adresów IP

Stare: ifconfig

ip addr (lub ip a)

Pokazuje fizyczne i wirtualne (veth/docker0) interfejsy wraz z maską (CIDR).

Tablica Routingu

Stare: route -n

ip route (lub ip r)

Gdzie są wysyłane pakiety? Szukaj default via 10.0.0.1 (Brama domyślna).

Tablica ARP

Stare: arp -a

ip neigh (lub ip n)

Kto (jaki MAC) odpowiada za dany adres IP w sieci lokalnej (L2).

Stworzono dla Inżynierów Systemów Rozproszonych.
