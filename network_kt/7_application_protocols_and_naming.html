Rozdział 7

Protokoły Aplikacyjne (L7)

Warstwa 7 to miejsce, gdzie abstrakcyjne pakiety zmieniają się w logiczne obiekty JSON, strumienie wideo i wywołania funkcji. Dowiesz się tutaj, jak zoptymalizować komunikację między serwerami, by nie marnować zasobów na zbędny narzut (overhead).

7.1. DNS w chmurze (Zabójczy Cache)

DNS (Domain Name System) tłumaczy przyjazne nazwy (np. api.users.local) na adresy IP. W systemach rozproszonych serwery bez przerwy wstają i padają, więc adresy IP przypisane do danej domeny zmieniają się bardzo często.

Pułapka DSE: Nieskończony Cache DNS w JVM

Mikroserwis A
Pytanie: "Jakie IP ma db.local?"
Odp: 10.0.0.5 (TTL: 60s)
Serwer DNS
Reakcja starej maszyny wirtualnej (np. Java 8):

"O, adres to 10.0.0.5! Zapiszę go sobie w pamięci RAM NA ZAWSZE, żeby więcej nie pytać serwera DNS i być szybszym. Zignoruję parametr TTL (Time To Live)."

Efekt na produkcji:

Baza danych na IP 10.0.0.5 ulega awarii. System Failover stawia nową bazę pod IP 10.0.0.8 i aktualizuje rekord DNS.

Wszystkie aplikacje Node.js czy Go łączą się z nową bazą. Twój mikroserwis w Javie dalej uparcie wysyła ruch na martwe 10.0.0.5, aż do ręcznego restartu kontenera.

Rozwiązanie: Zawsze weryfikuj ustawienia DNS Cache TTL w swoim języku programowania (w Javie jest to flaga networkaddress.cache.ttl w java.security). W środowisku Kubernetes / AWS ustaw tę wartość na max 10-30 sekund!

7.2. Ewolucja HTTP (Dlaczego REST bywa wolny)

Tradycyjne API RESTowe opierają się na protokole HTTP/1.1. Przez lata próbowano go "łatać" dodając nagłówki `Connection: Keep-Alive` (by nie robić Handshake'u TCP co request). Niestety, HTTP/1.1 ma fundamentalną wadę architektury.

HTTP/1.1 (Tekstowy i Sekwencyjny)

GET /api/users HTTP/1.1
Host: example.com
... to wszystko czysty tekst!
Połączenie TCP 1
Żądanie 1
Czeka na Odp 1

W HTTP/1.1 przez jedno połączenie TCP w danym momencie może przechodzić tylko jedno żądanie. Jeśli serwer przetwarza Żądanie 1 przez 2 sekundy, Żądanie 2 musi grzecznie czekać w kolejce, blokując przepustowość.

HTTP/2 (Binarny i Multipleksowany)

01001
11010
Nagłówki są kompresowane (HPACK) i binarne!
Połączenie TCP 1 (Strumienie)
Strumień A (Req1)
Strumień B (Req2)
Strumień A (Ciąg dalszy)
Strumień C (Req3)

Protokół kroi żądania na małe, binarne Ramki (Frames) przypisane do ID Strumienia. Ramki z różnych żądań mogą przeplatać się w locie przez to samo połączenie TCP. Koniec z blokowaniem kolejki!

A co z HTTP/3? Rozwiązuje problem gubienia pakietów przez TCP (HoL Blocking w warstwie L4), wykorzystując transport QUIC (UDP), o czym uczyliśmy się w rozdziale 6.

7.3. Protokoły RPC (Dlaczego gRPC wygrywa z REST)

W komunikacji między mikroserwisami wewnątrz klastra (Backend-to-Backend), REST over HTTP/1.1 z payloadem JSON to najgorszy możliwy wybór pod kątem wydajności. Standardem branżowym jest obecnie gRPC od Google.

Problem z JSONem (Serializacja)

// JSON (REST) - Duży, Tekstowy
{
  "id": 12345,
  "name": "Alice",
  "isActive": true
}
~50 bajtów + powolne parsowanie tekstu przez CPU
// Protobuf (gRPC) - Skompresowany, Binarny
08 39 12 05 41 6C 69 63 65 18 01
Klucze ("id", "name") są ukryte w schemacie (.proto), nie przesyłamy ich siecią!
~11 bajtów + natychmiastowe czytanie binarne

Pułapka na DSE: Load Balancing

gRPC wymaga protokołu HTTP/2 pod spodem. Zawsze.

Ponieważ HTTP/2 używa jednego, długiego połączenia TCP do przesyłania setek zapytań (Multipleksowanie), klasyczny L4 Load Balancer (np. w AWS / K8s Service) zawiedzie! L4 wyśle to jedno połączenie TCP do pierwszego Poda z brzegu, który przyjmie na siebie 100% ruchu, ignorując pozostałe 9 serwerów.

Aby gRPC dobrze się skalowało, musisz używać L7 Load Balancera (np. Envoy, Ingress, Istio), który otwiera pakiet, odczytuje ramki H2 i rozdziela pojedyncze żądania (strumienie) na różne Pody.

7.4. Komunikacja Asynchroniczna i Push

HTTP z założenia to protokół typu Żądanie-Odpowiedź. Co jednak, gdy potrzebujesz "wypychać" (Push) dane z serwera do klienta na żywo (np. powiadomienia, czat, giełda)? Mamy 3 główne podejścia.

1. Short Polling

Pytasz co 2 sekundy: "Są nowe dane?"

→ Req (Nie)
→ Req (Nie)
→ Req (TAK!)

Marnuje CPU i pożera porty ulotne (zły wybór).

2. Server-Sent Events (SSE)

Zwykłe HTTP. Serwer trzyma otwartą odpowiedź i "strzela" danymi.

→ Zwykły GET HTTP
← Dane (Event 1)
← Dane (Event 2)

Świetne do powiadomień i giełdy (One-way).

3. WebSockets

Pełny dupleks TCP. Obie strony wysyłają ramki kiedy chcą.

↔ Handshake (Upgrade to WSS)
← Ramka (Serwer)
→ Ramka (Klient)

Najlepsze dla Czatów i Gier Multiplayer (Two-way).

Stworzono dla Inżynierów Systemów Rozproszonych.
