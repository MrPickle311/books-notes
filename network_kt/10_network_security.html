Sieć wewnętrzna (VPC) nie jest już uważana za bezpieczną. W erze chmury szyfrujemy i uwierzytelniamy każdy pakiet, nawet wewnątrz własnego klastra.

Rozdział 10

Bezpieczeństwo Sieciowe

Kiedyś bezpieczeństwo polegało na postawieniu twardego Firewalla na brzegu sieci ("Zamek i Fosa"). Dziś napastnik często znajduje się już wewnątrz zamku (np. przez przejęty, podatny kontener). Dlatego nowoczesne systemy rozproszone buduje się w architekturze Zero Trust.

10.1. TLS/SSL i Podatek od Szyfrowania

TLS (Transport Layer Security) zapewnia trzy rzeczy: Poufność (szyfrowanie payloadu), Integralność (nikt nie zmienił danych po drodze) oraz Uwierzytelnienie (rozmawiasz z prawdziwym serwerem banku, a nie oszustem). Niestety, matematyka stojąca za tym procesem wymaga czasu i wielu podróży pakietów w tę i z powrotem (RTT).

Pełny koszt nawiązania bezpiecznego połączenia (TLS 1.2)

Klient Serwer
TCP SYN
← TCP SYN-ACK
Koniec L4 (Stracone 1 RTT)
ClientHello (Wybór szyfru) →
← ServerHello + Certyfikat publiczny
Wymiana Kluczy (Key Exchange) →
Koniec negocjacji (Stracone kolejne 2 RTT)
Zszyfrowane HTTP GET

Podatek od bezpieczeństwa

Zanim wyślesz pierwszego bajta danych do serwera np. w Australii (gdzie RTT wynosi 250ms), musisz poczekać 3 RTT (750ms!).

Uwaga: Nowszy protokół TLS 1.3 redukuje ten proces z 2 RTT do zaledwie 1 RTT, a dla wznawianych sesji do 0 RTT!

Dlaczego to jest wolne? (Asymetria vs Symetria)

Kryptografia asymetryczna (Klucz Publiczny/Prywatny - np. RSA) rozwiązuje problem zaufania, ale jest matematycznie niezwykle "ciężka" dla procesora. Dlatego TLS używa jej tylko na samym początku (podczas Handshake'u), aby bezpiecznie wygenerować i przekazać sobie "hasło". Po ustaleniu hasła, komunikacja przełącza się na błyskawiczną kryptografię symetryczną (np. AES), która szyfruje same dane (payload).

10.2. mTLS (Mutual TLS) i Zero Trust

W standardowym TLS, to Ty (Klient) sprawdzasz certyfikat Serwera, by upewnić się, że nie logujesz się na fałszywą stronę banku. Serwer banku jednak nie weryfikuje certyfikatów przeglądarki (weryfikuje hasło). W komunikacji między maszynami to nie wystarcza.

Uwierzytelnienie Obustronne (mTLS)

Serwis A (API)

Posiada Certyfikat:
CN=api-service
1. Serwis B weryfikuje certyfikat Serwisu A
2. Serwis A weryfikuje certyfikat Serwisu B

Serwis B (Baza)

Posiada Certyfikat:
CN=db-service
Jak to działa w Service Meshu?

Konfigurowanie i odnawianie tych certyfikatów ręcznie w kodzie każdej aplikacji to koszmar. W systemach chmurowych (np. Kubernetes z Istio), wirtualny Sidecar Proxy automatycznie przejmuje ruch, dorzuca swój certyfikat i weryfikuje certyfikat rozmówcy. Z perspektywy kodu Twojej aplikacji, wysyłasz zwykłe HTTP GET, a sieć pod spodem automatycznie zbroi to w mTLS. Jeśli napastnik uzyska dostęp do sieci VPC, nie podsłucha ruchu (szyfrowanie) i nie wyśle fałszywego zapytania (brak podpisanego certyfikatu mTLS).

10.3. Zapory: L4 Firewall vs L7 WAF

Zapory sieciowe chronią aplikację przed intruzami, ale tak jak Load Balancery, różnią się inteligencją (Warstwa 4 vs Warstwa 7).

Tradycyjny Firewall L3/L4 (Bramkarz)

  • Sprawdza tylko IP źródłowe i Port.
  • Ekstremalnie szybki (iptables, Security Groups w AWS).
  • Jeśli port 443 jest otwarty dla internetu, przepuści każdy zaszyfrowany ruch.
ALLOW IP 10.0.0.* PORT 443
DROP IP * PORT 22 (SSH)

WAF - Web App Firewall L7 (Celnik)

  • Deszyfruje ruch TLS i wczytuje się w treść zapytania HTTP.
  • Blokuje ataki SQL Injection, XSS, dziwne User-Agenty.
  • Wolniejszy. Wymaga regularnej aktualizacji reguł (np. modsecurity).
BLOCK IF Payload CONTAINS "SELECT * FROM"
BLOCK IF Header "User-Agent" IS NULL

10.4. Mechanizm CORS (The Frontend Nightmare)

Największe i najbardziej niezrozumiane kłamstwo sieci web: "Serwer blokuje mi żądanie przez CORS".

CORS (Cross-Origin Resource Sharing) to system bezpieczeństwa wbudowany w przeglądarki internetowe, a nie w serwery backendowe. Przeglądarka domyślnie zabrania skryptom na stronie strona-a.pl pobierania danych z API strona-b.pl (Same-Origin Policy), by złośliwa strona nie wykradła w tle danych z Twojego banku (gdy masz tam aktywną sesję/ciasteczka).

Kto tak naprawdę blokuje zapytanie?

Skrypt JS

fetch('api-b.pl')
Silnik Chrome
Czekaj! Odpowiedź z serwera NIE MA nagłówka:
Access-Control-Allow-Origin: strona-a.pl
api-b.pl

Twój Serwer

Przetwarza request, wysyła DB Query, zwraca 200 OK (ale bez nagłówków CORS).
Błąd w konsoli: CORS policy blocked...

Jak backendowiec naprawia CORS (Preflight OPTIONS)

Jeśli skrypt próbuje wysłać coś więcej niż prosty GET (np. POST z JSONem lub niestandardowymi nagłówkami), przeglądarka najpierw, bez wiedzy programisty frontendowego, wysyła tzw. Preflight Request metodą OPTIONS.

Pytanie przeglądarki (OPTIONS):
OPTIONS /api/users HTTP/1.1
Origin: https://strona-a.pl
Access-Control-Request-Method: POST
Poprawna odpowiedź Twojego API:
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://strona-a.pl
Access-Control-Allow-Methods: POST, GET, OPTIONS

Uwaga: Jeśli używasz Wildcarda Access-Control-Allow-Origin: *, przeglądarka odrzuci requesty, w których uwierzytelniasz się ciasteczkami (withCredentials=true) ze względów bezpieczeństwa.

Stworzono dla Inżynierów Systemów Rozproszonych.
