Rozdział 3

Infrastruktura: L3, Routing i SDN

Warstwa 2 (Switche) pozwalała komputerom rozmawiać w jednym biurze. Warstwa 3 (L3) łączy te biura w globalny Internet. To tutaj pakiety otrzymują adresy IP, a routery decydują, którą ścieżką w gąszczu światłowodów przesłać dane, by najszybciej dotarły do celu. Poznamy też zasady działania chmurowych VPC.

3.1. Adresacja IP i CIDR

Zapis CIDR (Classless Inter-Domain Routing), np. 10.0.0.0/24, określa rozmiar podsieci. Liczba po ukośniku to ilość bitów (z 32) zarezerwowanych na "część sieciową". Im wyższa liczba, tym mniejsza pula dostępnych adresów. Zrozumienie tego jest krytyczne przy projektowaniu klastrów.

RFC 1918 (Pule Prywatne)

  • 10.0.0.0/8 - (16.7 mln IP) Gigantyczne sieci korporacyjne.
  • 172.16.0.0/12 - (1 mln IP) Często domyślna pula w Dockerze.
  • 192.168.0.0/16 - (65 tys. IP) Typowe sieci domowe / małe biura.

Wyczerpanie puli IP w Kubernetes

K8s przydziela odrębne IP każdemu Podowi. Jeśli przydzielisz Node'owi pulę /24 (254 adresy), na tym serwerze uruchomisz max ~250 kontenerów. Jeśli ustawisz za małą pulę VPC, nagle nie będziesz mógł zeskalować aplikacji w chmurze!

Zajęte: 242/254 IP (VPC Subnet /24)

IPv6 w systemach rozproszonych

Przejście na IPv6 eliminuje wyczerpywanie adresów IP (oferuje 3.4 × 1038 adresów). Główne zmiany z perspektywy DSE:

  • Koniec z NAT: Każdy kontener i zasób dostaje unikalny, globalny adres IP. Odpada narzut procesora i pamięci na tablicę Conntrack oraz problemy z NAT timeout.
  • Brak fragmentacji w locie: Routery pośredniczące w IPv6 nigdy nie dzielą pakietów. Jeśli pakiet jest za duży, router go odrzuca i odsyła ICMP. Sprawne PMTUD jest w IPv6 krytyczne i obowiązkowe.
  • Dual-Stack: Systemy operacyjne obsługują oba stosy (IPv4 i IPv6) równolegle, wymagając odpowiednich ustawień socketów (np. IPV6_V6ONLY).

IP TTL vs DNS TTL: Częsty błąd pojęciowy

Skrót TTL (Time-To-Live) jest używany w obu warstwach, ale oznacza zupełnie inne mechanizmy:

Cecha IP TTL (Warstwa L3) DNS TTL (Warstwa L7)
Jednostka Licznik skoków (Hops) Sekundy (Czas)
Działanie Dekrementowany o 1 przez każdy router. Zapobiega krążeniu pakietu w pętli w nieskończoność. Czas, przez jaki resolver DNS (np. przeglądarka) może trzymać w pamięci podręcznej wynik zapytania.

3.2. SDN i Wirtualne Sieci Chmurowe (VPC)

W AWS (VPC) czy GCP, nie masz dostępu do fizycznych switchy i routerów. Wszystko to SDN (Software-Defined Networking). Kernel hypervisora używa reguł Match-and-Action (Dopasuj i Wykonaj), by "udawać" sprzęt sieciowy w pamięci RAM. Zmiana routingu to tylko aktualizacja rekordu w bazie danych chmury.

Logika Match-and-Action w SDN

Reguła 1 Match: Dst IP 10.0.1.0/24
Action: Przekaż do Subnet-A
Reguła 2 Match: Dst IP 0.0.0.0/0 (Internet)
Action: Przekaż do Internet Gateway (IGW)
Reguła 3 (Security Group) Match: Port 22 (SSH)
Action: DROP (Odrzuć)

3.3. BGP i Magia IP-Anycast

BGP (Border Gateway Protocol) to system nerwowy internetu. Routery BGP wymieniają się informacjami typu "Mój system potrafi dowieźć ruch do sieci 1.0.0.0/8". Dzięki BGP możemy zbudować najpotężniejszy wzorzec systemów globalnych: IP-Anycast.

Jak działa Anycast (np. Cloudflare 1.1.1.1)?

User (EU)
User (USA)
Data Center (EU)
IP: 1.1.1.1
Data Center (USA)
IP: 1.1.1.1

Dlaczego to jest Game Changer?

Wielu serwerom na świecie przypisujesz dokładnie ten sam adres IP. BGP automatycznie ogłasza te trasy do internetu. Gdy użytkownik z Europy wpisuje ten adres IP, routery szkieletowe wybiorą najkrótszą drogę – czyli do serwera w Europie. Gdy serwer w Europie padnie, BGP to wykryje i automatycznie przekieruje ruch z Europy do USA. Tak buduje się globalne CDN-y, serwery DNS i systemy o zerowym czasie przestoju (Zero Downtime).

3.4. BFD (Szybkie Awarie)

Protokoły routingu (jak BGP czy OSPF) są dość powolne. Jeśli zgasimy serwer "twardo", BGP może potrzebować od kilkunastu sekund do kilku minut (tzw. Hold Timer), żeby zorientować się, że serwer nie żyje.

Rozwiązanie: BFD (Bidirectional Forwarding Detection)

BFD to ultraszybki, "chudy" protokół. Wysyła pakiety kontrolne (hello) co np. 50 milisekund! Gdy zgubi 3 takie pakiety pod rząd (150ms), krzyczy do powolnego BGP: "Ten link nie żyje, natychmiast przelicz trasę!".

Dzięki temu, awaria światłowodu jest obsługiwana w czasie niezauważalnym dla większości połączeń TCP (gra multiplayer nawet nie zerwie sesji).

3.5. VRF (Wirtualny Routing)

VRF (Virtual Routing and Forwarding) to dla routerów (L3) tym, czym VLAN-y dla switchy (L2). Pozwala on na stworzenie wielu, całkowicie odizolowanych tablic routingu wewnątrz jednego fizycznego routera.

Jeden Hardware, Dwie Firmy

VRF: Tenant A
Sieć: 10.0.0.0/16
Brama: 10.0.0.1
VRF: Tenant B
Sieć: 10.0.0.0/16
Brama: 10.0.0.1

Dzięki VRF, różni klienci chmury (Tenants) mogą używać dokładnie tych samych adresów IP, a ruch nigdy się nie pomiesza.

3.6. NAT, Middleboxy i Timeouty gRPC

NAT (Network Address Translation) pozwala ukryć całą prywatną sieć pod jednym publicznym adresem IP. Kiedy wysyłasz zapytanie z AWS w internet, NAT Gateway (Middlebox) modyfikuje Twój nagłówek IP w locie, zapamiętując to w swojej wewnętrznej tabeli (ang. Connection Tracking / conntrack).

App
10.0.0.5:1234

NAT Gateway
Tablica (Conntrack):
10.0.0.5:1234 →
8.8.8.8:40500
Internet API
Widzi ruch od:
8.8.8.8:40500

Pułapka DSE: NAT Idle Timeout

Routery NAT mają ograniczoną pamięć RAM na tablicę połączeń (Conntrack). Jeśli przez dane połączenie TCP nic nie przepłynie przez np. 5-10 minut (zależy od chmury/routera), NAT cicho usunie ten wpis (NAT Drop).

Dlaczego gRPC i bazy danych "zawisają"?

Protokoły takie jak gRPC czy połączenia z bazą SQL (Connection Pooling) utrzymują długie, trwałe połączenia TCP. Jeśli system jest bezczynny w nocy i nic nie wysyła, NAT ucina połączenie w połowie drogi. Twój mikroserwis i Baza Danych nadal myślą, że połączenie żyje. Gdy rano przychodzi zapytanie klienta, aplikacja próbuje wysłać dane w martwy kanał – i wisi, aż nastąpi 15-minutowy Timeout TCP poziomu jądra.

Rozwiązanie: Włączenie mechanizmów KeepAlive na poziomie aplikacji (gRPC KeepAlive, TCP KeepAlive w sterowniku bazy). Sztuczne wysyłanie "pustego" pingu co 1-2 minuty, by odświeżyć czas życia (TTL) w tablicy routera NAT!
