# Chapter 24: Threat Modeling Applications

Threat modeling identifies threats, threat actors, existing mitigations, and the delta between threats and mitigations. A well-implemented model serves as a living forward knowledge repository for handing off security knowledge.

## Designing an Effective Threat Model

A proper threat model must accomplish the following goals:
1.  **Document knowledge**
2.  **Identify threat actors**
3.  **Identify risks** (attack vectors)
4.  **Identify mitigations**
5.  **Identify delta** (between risks and mitigation)

Risks are not actionable until existing mitigations are known.

## Logic Design

-   **What it is:** A description of a feature/product from a functionality perspective (higher-level than technical architecture, lower-level than marketing).
-   **Why it is important:** Used to spot **logic vulnerabilities** (vulnerabilities derived from the application not following its intended logic/business requirements, rather than traditional application-level flaws).
-   **Example Vulnerability:** Bypassing a web form to post a review score > 5, skewing aggregated metrics.

## Technical Design

-   **What it is:** Documentation of the implementation details of a product/feature to identify threats generated by the architecture.
-   **Key Variables to Identify:**
    -   Tools and technologies (languages, databases, build tools, package managers, frameworks).
    -   Third-party services (in-network like AWS S3; out-of-network like Datadog, CircleCI).
    -   Data flow (data flow diagrams [DFD], data formats in transit, encryption in transit, network protocols).
    -   Network configuration (firewalls, VPNs, physical networks).
    -   Authentication and authorization controls (multiple token types, API endpoint checks).
    -   Database schema (regulated/controlled data, NoSQL data shapes).
-   **How it works:** Cross-referencing technical specifics with the logic design exposes traditional vulnerabilities (e.g., XSS, SQLi, Information Disclosure).
-   **When to use:** In conjunction with logic design prior to brainstorming vulnerabilities.

## Threat Identification (Threat Actors)

Threat actors are archetypical attackers with unique routes for attacking a system.
-   Must include internal, external, and machine users (e.g., autonomous internal scripts).
-   Must split into authenticated and unauthenticated permission sets.
-   Each threat actor has a varying attack surface area.

### Table 24-1. Potential Threat Actors

| Threat Actor | Powers, Permissions, and Risk |
| :--- | :--- |
| **User admin** | Can use admin UI to read/update DB. May steal PII or modify ratings. High risk if logging is improper. |
| **Customer support user** | Can read DB data but not update fields. Could steal PII. |
| **Review aggregator script** | Machine user with DB admin access. Significant damage potential if compromised. |
| **Authenticated user** | Can post reviews. Could attempt malicious payloads (SQLi, overflow). |
| **Unauthenticated (guest) user** | Can read scores. Low threat (barring DoS/circular queries). |

## Threat Identification (Attack Vectors)

-   **Attack Vectors:** Potential (future) pathways by which a threat actor could attack your system.
-   Best brainstormed via cross-analysis of logic design, technical design, and threat actors.
-   Include severity rankings (e.g., P0–P4) to prioritize work.

### Table 24-2. Attack Vectors (Example)

| Threat Name | Severity | Threat Actor | Description |
| :--- | :--- | :--- | :--- |
| Improper validation | P1 | All users except guest | Score out of 0-5 bounds leading to skewed ratings. |
| Information disclosure | P3 | All users | Error messages leaking unreleased features. |
| SQL injection | P0 | All users except guest | POST payload leading to SQLi. |
| GraphQL circular queries | P1 | All users | Large self-referential queries causing DoS. |
| GraphQL introspection | P1 | All users | Leaking server configuration details. |
| High privilege user attacks | P0 | Admin & aggregator script | Compromised privileged token compromising system. |

## Identifying Mitigations

-   Data collection phase to document existing security controls mitigating the identified attack vectors.

### Table 24-3. Existing Mitigations

| Mitigation Name | Attack Vectors Mitigated | Description |
| :--- | :--- | :--- |
| Validation logic | Improper validation | Engine rejects scores outside 0-5. |
| SQL Injection Mitigations | SQL injection | Use of DSL forcing prepared statements. |

## Delta Identification

-   Cross-reference attack vectors with the mitigation list, removing those with sufficient mitigations.
-   The remaining vectors form the **Delta**.
-   Brainstorm new mitigations for the unmitigated attack vectors.

### Table 24-4. Delta Identification (Unmitigated Vectors)

| Threat Name | Severity | Threat Actor | Description |
| :--- | :--- | :--- | :--- |
| Information disclosure | P3 | All users | Error messages leak unreleased features. |
| GraphQL circular queries | P1 | All users | Missing rate/compute limits. |
| GraphQL introspection | P1 | All users | Introspection enabled, leaking config. |
| High privilege user attacks | P0 | Admin & aggregator script | Lack of monitoring or token scoping. |

### Table 24-5. Mitigations for Identified Attack Vectors

| Mitigation Name | Attack Vector Mitigated | Description |
| :--- | :--- | :--- |
| Better error messages | Information disclosure | Generic error messages. |
| GraphQL compute limits | GraphQL circular queries | Implement max compute time for queries. |
| GraphQL configuration | GraphQL introspection | Disable introspection and suppress internal errors. |
| Privileged permissions rework | High privilege user attacks | Scope tokens to required functionality, log API calls off-platform, restrict script DB permissions. |

-   **Next Steps:** Mark off each mitigation as it is implemented. Once all mitigations are implemented, the immediate purpose is complete. If the feature scope increases, the threat model must be updated.

## Summary

-   Threat models are long-lived, highly effective sources of centralized technical and security knowledge.
-   They identify attackers and vulnerabilities before the application is developed.
-   Refining security posture early in the development phase is highly cost-effective ("10 hours of fixes in the architecture phase is worth 100 hours of fixes postlaunch").
